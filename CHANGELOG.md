## [0.4-pre.5] - 2026-06-09

**Judge 校准脚本就位**——任何 LLM-as-Judge 上线前必经的 r/Kappa ≥ 0.7 验证。

### 新增

- **tools/judge_calibration.py**（678 行）· 跨 archetype 通用的校准脚本
  - 生成型 → 每维度 Pearson r（人工算 vs 加权算）
  - 分类型 → Cohen's Kappa + linear-weighted Kappa（处理 ordinal）
  - hand-rolled 数学（不依赖 scipy / numpy）
  - 单元测试覆盖：完美相关 / 反相关 / 零相关 / Kappa 完美 / Kappa 负
  - 输出：人读 markdown 报告 + 机读 JSON summary
  - `--threshold` 默认 0.7（来自 EVAL_TAXONOMY §4.3）
  - 不达标时给具体诊断（哪个维度偏 / over-rate vs under-rate / 最常见的 disagreement pattern）
  - 分类型对"high 漏检"加 ⛔ critical 警告（隐私优先）

- **tools/examples/human_eval_L3_25.example.jsonl** · 3 条生成型 human-eval 模板
  - 含好例（hint_001 高分）+ 中例（hint_009 标准）+ 反例（合成的 hint_032_synthetic_bad，让远亲借钱直接答应——human 给 ~2 分，验证 judge 也能给低分）
- **tools/examples/human_eval_L3_49.example.jsonl** · 4 条分类型 human-eval 模板
  - 含完美一致（sens_001）+ 模型漏检（sens_034 把 high 标 medium）+ 模型误报（sens_046 把减肥标敏感）+ 谐音陷阱（sens_065 HTML 当 HIV）

- **tools/README.md**（300+ 行）· 完整使用指南
  - 为什么必须做（错位 judge → 错决策 → 烧信任）
  - 7 步验证流程
  - 两种 archetype 的命令行示例
  - human-eval JSON schema
  - 已知 bias + 重校准时机
  - 数学细节（Pearson / Cohen / linear-weighted Kappa）

### 设计要点

- **同一脚本两种 archetype**：用 `--archetype generation|classification` 切换
- **不依赖 scipy**：hand-roll 数学，单文件即可跑
- **保守原则贯穿**：分类型出 verdict 时，"high 漏检"特别警告（隐私阻断器逻辑）
- **诊断驱动**：不达标时报告自动给"修哪个维度 / 怎么修"建议

### 验证

- ✅ Python AST clean（678 lines）
- ✅ Pearson r 数学单测：完美 = 1.0 / 反相关 = -1.0 / 常数 = 0.0
- ✅ Cohen's Kappa 单测：完美一致 = 1.0 / 完全冲突 < 0
- ✅ 两个 archetype 的 human-eval example 都 schema-valid

### 部署 judge 的强制流程（现在）

```
建 judge prompt
   ↓
建 30+ 条 human-eval（3 人独立标注，IAA Kappa ≥ 0.7）
   ↓
跑 tools/judge_calibration.py
   ↓
✅ DEPLOYABLE → judge 可进 evaluate.py / evaluate_L3_49.py
❌ NOT DEPLOYABLE → 修 judge prompt 重做（最多 3 次）
                  → 仍不达标 → 该能力降级到 C 档（必须人工）
```

**第二个 eval archetype 建好**——分类型能力的参考实现。和 L3.25（生成型）一起，覆盖了 52 个 L3 叶子的两大形态。

    - 30 条 low（含 hard negatives：化工 / HTML / 心理学 / 减肥 / 戒咖啡因 / 红酒 / 法考——专门挑选谐音 / 字面相似但实际无害的样本）
    - 覆盖中英混用 / 拼音缩写（dy zheng / fq / hd）/ emoji 模糊 / 黑话（黑狗 / 糖友 / 上岸）/ 长句委婉
  - `evals/judges/L3_49_sensitive_mainline_check.md` · 仅 disagreement 用的 judge prompt
  - `evals/schemas/L3_49_io.json` · 含 v0.7-alpha + v1.0-general 两版 launch_gates
  - `evals/pipeline/evaluate_L3_49.py` · **673 行**分类型 archetype runner
  - `evals/README.md` 改写为双 archetype 版本

### Pipeline 与 L3.25 的关键差异

| 维度 | L3.25 生成型 | L3.49 分类型 |
|------|-------------|-------------|
| 主评测 | LLM-as-Judge 5 维度 1-5 分加权 | Confusion matrix（程序化）|
| Judge 调用 | 每条都调 | 仅 disagreement 调用（省钱）|
| Launch gate | avg ≥ 3.75 / 5（平均阈）| recall_on_high ≥ 99%（绝对阈）|
| 报告关注 | worst-10 + 维度均值 | False negatives + per-category 召回 |
| 关注 bias | 长度偏好 / 同模型偏好 | 隐私保守原则 / 同语言偏好 |

### 设计意图

L3.49 是**分类型 archetype 参考**——其他分类能力（L3.20 优先级 / L3.24 意图 / L3.51 数据分类 / L3.52 PII）照这个文件 copy + specialize。L3.25 是生成型 archetype 参考——其他生成能力（L3.26 草稿 / L3.34 推荐 / L3.39 复盘反馈）照那个 copy。

两个 archetype 一起覆盖了 52 个 L3 叶子的全部形态。

### 数据集设计要点

L3.49 的 hard negatives 是这个 dataset 最珍贵的部分。系统**最容易在这些场景误报**：

- "化工材料学习" → 化疗谐音陷阱
- "学 HTML/CSS" → HIV 谐音陷阱
- "戒咖啡因" → 看似 addiction 实则 low
- "学心理学" / "考心理咨询师" → 看似 mental_health 实则 low
- "看《亲密关系》" → 看似 relationship_crisis 实则 low
- "学投资理财" → 看似 financial 实则 low

如果模型把这些误报为敏感，用户会感觉"正常人生目标都被算法当病人对待" → 信任崩塌。

- ✅ Python AST clean（673 lines）
- ✅ 80 条 Gold 全部 schema-valid
- ✅ 所有 high 条目 user_override_allowed = false（绝对约束）
- ✅ 所有 hard_negative tag 的条目都标 low（trap 检查通过）
- ✅ Judge prompt 解析 OK，3 个 placeholder 都在

### Pipeline 特性

- **三档 eval 集成**：A 程序化（schema 校验）+ B LLM-judge（5 维度评分）+ C 抽样人工

- **报告自动生成**：含 summary / 维度均值 / 上线 gate / worst-10 / 下一步建议
- **抗 bias 设计**：被测匿名 + 多裁判 + 抽样人工对照 + 校准 r ≥ 0.7

### 设计意图

L3.25 是**参考实现**——其他 51 个 L3 能力照这个模式 copy + specialize：
1. 改 candidate prompt 适配该能力
2. 写新的 dataset / judge / schema 三个文件
3. 跑同一个 evaluate.py（仅改路径参数）

### 文件大小

- evaluate.py：634 行（30 分钟可读完，复杂工程化推迟到 v0.5）

## [0.3.0] - 2026-06-04

让小 P 从「分析者」升级为「共谋者」，并把学习信号显性化。

### 新增

- **小 P 给出明确推荐 +「采用 / 辩论」操作**
  - 在「跟小 P 商量」面板的最后一个气泡，小 P 不再只是分析，而是给出明确建议（带 ⭐ 图标 + 高/中把握度标签）
  - 每位联系人的推荐都包含：
    - `verdict`：推荐哪个 hint（如「热情确认」「承诺时间」「主动陪同」）
    - `reasoning`：3 条具体理由
    - `agreedNote`：用户已选了这个 → 小 P 给细微优化建议
    - `disagreedNote`：用户选了别的 → 小 P 用黄色提示警告 + 折中方案
    - `debate`：用户点辩论 → 小 P 反问"你哪里不确定"
  - 「采用」按钮 → 自动选中推荐的 hint + 关闭面板 + toast 提示
  - 「辩论一下」按钮 → 小 P 在面板内继续追问，按钮置灰，输入框 focus

- **小 P 学习信号（pi-insight）**
  - 紫色渐变浮层，从顶部滑入，停留 5.5s 自动消失
  - 关键时刻自动触发，让用户感知"小 P 真的在学"：
    - 用户采纳小 P 推荐 → 「这次的成功反馈我也学进去了」
    - 用户连续点「更短」chip → 「我开始相信偏爱简短是你的真实偏好」
    - 用户对家人点「更暖」chip → 「我把这个偏好升级为高置信度」
    - 用户连续发送 → 「你的语言指纹我又清晰了一点」
    - 进入 Noa 这种高难度场景 → 「这类对话的 chip 排序我调整了」
  - 与普通 toast 视觉区分：紫色渐变 + π avatar + "小 P 学到了" 标签

### 改动

- 移除 openPiContext 中原本"基于 hint 的简单 pivot 文案"，改为基于 recommend 数据的结构化推荐
- 4 个学习触发函数：maybeLearnFromSend / Adopt / Chip / Contact

### 设计原则记录

人格化 AI 的进阶要求：
1. **从分析者到共谋者**——光给数据不够，要敢于给意见、敢于被反驳
2. **学习要"被看见"**——AI 在悄悄学习不会建立信任，要把"我学到了 X"显性化（同时给「忘掉」入口形成闭环）
3. **不同意时也要给路径**——不是"你错了"而是"你这样选风险是 X，建议至少补一句 Y"

## [0.1.0] - 2026-06-02

首个公开 demo 版本，覆盖 4 大核心模块的完整体验闭环：智能社交收件箱、聊天 + AI 副驾驶、人物档案、关系网络图谱。

修复概念重叠和品牌一致性问题：合并「AI 分析」和「跟小 P 商量」、统一去掉 user-facing 的「AI」称呼。

### 改动

- **合并「AI 分析」与「跟小 P 商量」为单一入口**
  - 草稿区下方原有两个按钮（结构化分析 + Pi 对话），功能高度重叠
  - 现在合并为一个按钮：「跟小 P 商量这条 · 看历史 + 追问」
  - 弹起的 sheet 由 Pi 用她自己的语气把结构化分析说出来：
    - 「她到底想说什么」（intent）
    - 「她真正期待」（expectations）
    - 「历史里值得看的」（history bullets）
    - 「⚠️ 风险点 + 你倾向哪个？」（pivot to chat）
  - 然后无缝衔接到 chips + 自由追问

- **统一 user-facing 品牌：「AI」→「小 P」**
  - 「AI 早间简报」→「小 P · 早间简报」（v0.2 已改）
  - 「AI 已分析」→「小 P 看完了」
  - 「AI 推荐：附和」→「小 P 推荐：附和」
  - 「AI 仅在本地」→「小 P 仅在本地」
  - 「AI 改写」→「小 P 改写」
  - 「AI 已总结」→「小 P 已总结」
  - 「AI 优先级 / AI 已为你过滤 / AI 备忘」全部改为小 P
  - 「Pulse · AI」badge → 「Pulse · 你的副驾驶」
  - title 也从「Pulse · AI 社交副驾驶」改为「Pulse · 小 P 社交副驾驶」

- **保留「AI」的唯一例外**
  - Mira 档案中「她对 'AI 自动回复' 持保留态度」——这是内容（角色对 AI 工具的看法），不是产品功能描述。保留这个细节作为产品叙事的一部分（暗示用户使用 AI 副驾驶时也需要小心被对方察觉）。

### 移除

- `#analysis-sheet` HTML 结构（被 #pi-context-sheet 取代）
- `openDeepAnalysis()` / `closeDeepAnalysis()` / `askAnalysis()` JS 函数（功能被 `openPiContext` / `askPiContext` 吸收）

### 设计原则记录

人格化的核心要求：**有了名字，就不能再用通用名词**。否则同一个 AI 在用户面前有两个"主语"，就像有个朋友叫小李，你跟别人说话时一会儿叫他「小李」一会儿叫「那个人」——会破坏关系连续性。

## [0.2.0] - 2026-06-03

引入 AI 人格「小 P / Pi」，把 AI 从功能升级为有名字、可对话、可纠正的副驾驶角色。

### 新增

- **小 P（Pi）作为 AI 人格**
  - 独立的紫色渐变圆形 + 数学符号 π 视觉标识
  - 替换底部 tabbar 第 3 位「我」入口，带脉冲发光动效
  - 状态栏：「在线 · 仅本地运行 · 已为你工作 12 天」

- **小 P 主面板 4 个模块**
  1. **对话**：自由对话窗口 + 4 个预设快捷指令（"以后回客户少用 emoji"等）
  2. **风格**：默认语气/Emoji/直接度/长度 4 个三档滑块 + 主动行为开关
  3. **权限**：读取权限/行为权限/数据存储 三组开关，含锁定的系统级承诺
  4. **已学到**：5 项偏好可视化（带置信度进度条），可「忘掉」

- **时间感知的开场**（v0.2 新增）
  - 早上：昨晚未看的消息 + 今天概览
  - 中午：上午处理量 + 关系信号
  - 下午：当前最紧急的消息提醒
  - 晚上：今日总结 + 明日预告
  - 深夜：建议早些休息 + 已默默处理的群消息

- **「忘掉」需要解释**（v0.2 新增）
  - 点击「忘掉」打开底部弹窗，要求选择原因
  - 4 种预设原因：不准确 / 已过时 / 不希望被推断 / 其他
  - 小 P 根据原因给出不同回应（在主对话流追加）
  - 卡片淡出动画，并在 Pi 主对话里留下记录

- **聊天页召唤小 P**（v0.2 新增，v0.2.1 合并优化）
  - 在草稿区的「AI 分析」按钮旁新增「💬 跟小 P 商量」（v0.2）
  - v0.2.1 合并为单一入口

- **针对人群的风格覆盖**（v0.2 新增）
  - 在风格设置底部新增「针对人群单独设置」section
  - 4 个关系类型卡片：家人 / 朋友 / 同事 / 客户
  - 点击进入该群组的覆盖编辑器（独立 sliders + 专属 toggles）
  - 例：客户 = 简练 + 不用 emoji + 称呼用「您」+ 承诺时附 timeline

### 改动

- 收件箱顶部 AI 摘要署名改为「小 P · 早间简报」
- 侧边栏 demo 介绍更新为以小 P 为主角
- 体验路径提示新增第 6 步「底部 tabbar → 小 P」

## [0.1.0] - 2026-06-02

首个公开 demo 版本，覆盖产品 4 大核心模块的完整体验闭环。

### 新增

- **智能社交收件箱**：按 AI 推荐的优先级（立刻回 / 稍后回 / 可忽略）分组，替代时间排序
- **AI 早间简报**：顶部紫色卡片，提示当日待处理消息和被忽略的人
- **聊天 + AI 副驾驶面板**：基于关系和对方话语动态生成 2 个 hint，配合每 hint 的场景化快捷调整 chips
- **AI 深度分析浮层**：从底部弹起的全功能 sheet，包含表层意图、潜在期待、关联历史上下文（带具体日期）、风险点，以及可继续追问的对话区
- **人物档案**：三 Tab 切换（概览 / 观点档案 / 时间线），含关系热度可视化和 AI 备忘
- **关系网络图谱**：SVG 同心圆布局，按关系强度分 3 圈，按社群分 4 色，包含「久未联系」脉冲动画
- **关系健康度提醒**：底部卡片，自动检出需要关注的关系（妈妈 30 天未联系等）
- **微信形似 + Pulse 紫色 AI 标识**：聊天气泡严格还原微信视觉，紫色专属于 AI 元素

### 体验设计的关键迭代（v0.1 session）

1. **从「四态固定按钮」改为「2 个动态 hint」**
2. **保留并强化「AI 分析」入口**（深度面板 + 历史 + 可追问）
3. **加入「场景化快捷调整 chips」作为草稿改写的主路径**
4. **自定义指令默认可见而非折叠**

### 部署

- GitHub Pages: https://linhkhaiphung3680-prog.github.io/pulse-demo/
- 单 HTML 文件，含 Tailwind CDN，无需后端
