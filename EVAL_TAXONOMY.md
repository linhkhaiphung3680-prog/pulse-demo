## 0. 这份文档解决什么问题

我们正在从 v0.3 demo（Wizard of Oz · 关键词匹配）走向 v0.4 真 LLM 接入。一旦切换，就要回答三个问题：

1. **怎么知道 AI 没出错？** —— 没有 eval 系统，每次模型调整都是赌博
2. **谁来生产足够的数据？** —— 没有数据生产策略，eval 没有 ground truth 可对照
3. **多少分才能上线？** —— 没有门禁标准，"差不多了"会一直差不多

这份文档是**一个统一的 source of truth**：

- 把 Pulse 全部 AI 能力切到**叶节点级别**，每个叶子是一个独立可测的能力
- 给每个叶子定义：输入 / 输出 / 理想态 / 量化指标 / 反例 / eval 方法 / 数据生产
- 给团队明确的协作合同——
  - **AI 工程**：你的输出要满足这个 schema 和这些 metric
  - **数据生产**：你要按这些来源 / 数量 / 标注策略产数据
  - **模型评测**：你照这个跑 eval、按这个聚合分数、按这个判断能否上线

### 0.1 怎么用这份文档

- **找一个 AI 能力的全部信息**：去 §3，按分类树定位到叶子
- **生产数据**：去 §4 看每个叶子的"数据生产"小节
- **跑 eval**：去 §1 看方法论，去 §3 拿每个叶子的 metric 和理想态
- **判断能不能发布**：去 §5 看门禁

## 1. 评测方法论：三档 eval

每个 AI 能力都对应 1-3 档评测，按成本 / 准确度 / 可扩展性平衡：

### 1.1 档位 A · 程序化评测（Programmatic Eval）

- **谁评**：自动化脚本
- **怎么评**：精确 / 启发式匹配
- **优点**：0 成本、毫秒级、可大规模回归
- **限制**：只适合"有标准答案"的能力（schema 校验、关键词命中、阈值判断）
- **典型 metric**：accuracy / precision / recall / F1 / schema validity

**适用能力**：意图分类、关系类型识别、Hint slot 填充正确性、JSON schema 合规、敏感词检测、PII 泄漏检测

### 1.2 档位 B · LLM-as-Judge（模型裁判）

- **谁评**：用一个**更强**或**专精**的模型当裁判（推荐 Claude Sonnet/Opus 跑 judge）
- **怎么评**：给定 prompt、被测模型输出、评测维度，让裁判按 1-5 / 1-7 / 1-10 打分
- **优点**：可大规模、可处理"主观"任务
- **限制**：裁判本身有 bias（同模型偏好、长度偏好、表面流畅性偏好）
- **关键设计**：
  - 多裁判（≥ 2 个不同家族模型）取平均
  - 双盲（不告诉裁判哪个是被测模型）
  - 与人工评测 sample 对照（≥ 100 条）确保 judge-human 相关性 ≥ 0.7
- **典型 metric**：Likert score / pair-wise win rate

**适用能力**：草稿质量、Pi 推荐说服力、复盘反馈贴切度、人格一致性、共情程度

### 1.3 档位 C · 人工评测（Human Eval）

- **谁评**：内部团队 / 标注员 / 真实用户
- **怎么评**：双盲、按 rubric 打分 / pair-wise 对比 / 真实使用追踪
- **优点**：最接近真实用户体验
- **限制**：贵、慢、规模小
- **典型 metric**：5-point rating / win rate / NPS / 真实采纳率

**适用能力**：北极星陈述质量、复盘真诚度、同路人匹配的"被理解感"、长期偏好图谱准确度

### 1.4 档位选择决策树

```
能力是否可写出确定的 ground truth？
├─ 是 → 档位 A（程序化）
└─ 否 → 是否有训练好的 judge？
    ├─ 是 → 档位 B（LLM-as-Judge）+ 抽样人工对照
    └─ 否 → 档位 C（人工）
```

**默认组合**：每个 AI 能力同时跑 A + B 自动化，用 C 抽样对照（5% 样本）。

## 3. 一级能力详解

> 每个叶子能力的格式：
> - **任务**：一句话描述
> - **输入 / 输出**：schema
> - **理想态**：perfect 输出长什么样（含正例 + 反例）
> - **量化指标**：metric + 目标值
> - **eval 方法**：A / B / C 哪一档
> - **数据生产**：来源 + 数量目标 + 标注策略

---

### L1.1 身份与主线（Identity & Mainline）

#### L3.1 · 首启 onboarding 引导写北极星

**任务**：在用户首次打开 App 时，引导他写出 ≤ 200 字的身份陈述（不是从清单里选）。

**输入**：
- 用户的设备语言、时区、注册渠道
- 0 行历史数据（首次打开）

**输出**：
```typescript
{
  promptText: string,         // 给用户的引导问题
  examples: string[],         // 3 个范例
  delayOption: string,        // 「先用 2 周再答」按钮文案
  validationRule: {
    minLen: 8, maxLen: 200,
    rejectPatterns: ["更好的自己", "成长", "成功"]   // 太空泛拒收
  }
}
```

**理想态**：
- ✅ 范例本身**带个性**："我想成为我父亲那样有耐心、不发火的丈夫"
- ✅ 引导问题让用户**愿意停 5 秒思考**："想象 5 年后，你希望谁会在你的葬礼上说什么？"
- ✅ "先用 2 周再答"按钮显眼，不让人觉得跳过 = 失败
- ❌ 反例：范例是"成为更好的自己"（鸡汤）/ 引导问题"你的目标是什么"（KPI 语言）

**量化指标**：
- 首启 30 天内完成北极星比例 ≥ 50%（含先用 2 周用户）
- 北极星陈述平均长度 30-100 字（太短 = 模板，太长 = 编造）
- "先用 2 周"用户 14 天后再设定的转化率 ≥ 60%

**eval 方法**：B（LLM judge 评质量）+ C（5% 真实用户人工评）

**数据生产**：
- 合成：用 50 个 persona × 5 个表达风格 = 250 个理想北极星样例
- 真实：内测用户的实际北极星（脱敏后）
- 标注策略：3 人独立标"模板感 0-5"，取众数

#### L3.2 · 「先用 2 周再答」延迟决策

**任务**：用户跳过北极星后，小 P 在 14 天内观察对话 / 行为 / 复盘，第 15 天起草一份候选北极星给用户审。

**输入**：
- 14 天的所有对话（脱敏后的 entity / topic / sentiment 抽取）
- 14 天的复盘记录
- 用户标记过的"重要"联系人 / 话题

**输出**：
```typescript
{
  draftedNorthStar: string,   // ≤ 200 字
  evidence: [
    { source: "对话 #1234", excerpt: "...", relevance: 0.85 },
    { source: "复盘 #45", excerpt: "...", relevance: 0.78 }
  ],
  confidence: 0.65,           // 0-1
  alternatives: string[]       // 2-3 个候选给用户选
}
```

**理想态**：
- ✅ 候选北极星**用用户自己的话**（引用过他写过的关键短语）
- ✅ 至少 5 条 evidence 支撑，跨 ≥ 3 个不同来源
- ✅ 用户感觉"是的，这就是我"——而非"AI 觉得我应该是这样"
- ❌ 反例：通用化候选（"成为更专业的人"）/ 仅基于 1 个来源 / 没有 evidence

**量化指标**：
- 用户接受候选率 ≥ 40%（直接用 + 微调后用）
- evidence 引用准确率 ≥ 90%（人工抽检无幻觉）
- 候选与用户最终北极星 ROUGE-L ≥ 0.4

**eval 方法**：C（人工评 quality + evidence 准确性）

**数据生产**：
- 合成：构造 30 个虚拟用户的 14 天对话流，由 3 人为每个用户写"理想北极星"
- 真实：v0.4 alpha 内测里跳过北极星的用户的真实候选（人工标注接受/拒绝）

#### L3.3 · 30 天观察后改写建议

**任务**：用户已设定北极星但已用 30 天，小 P 基于行为数据建议是否改写。

**输入**：
- 当前北极星
- 30 天对话 + 复盘 + 主线动作历史
- 用户最常说的关键短语 top 10

**输出**：
```typescript
{
  shouldSuggestRevision: boolean,
  reason?: "drift_detected" | "user_words_more_specific" | "stale",
  suggestedText?: string,
  diff: { added: string[], removed: string[], kept: string[] },
  confidence: 0.7
}
```

**理想态**：
- ✅ 仅在用户实际行为已偏离北极星 ≥ 30 天才建议（不要每天烦扰）
- ✅ 修订建议保留用户原意 70%+，只调整 30% 让它更精准
- ✅ 显示"删 / 加 / 保留" diff，让用户能直接对照
- ❌ 反例：每周都建议改 / 改写后失去原意 / 没说为什么要改

**量化指标**：
- 建议触发率（在 30 天用户里）≤ 20%（不要乱触发）
- 用户接受改写率 ≥ 35%
- 改写后北极星与原版语义相似度 0.5-0.85（既保留又改进）

**eval 方法**：B + C

**数据生产**：
- 真实：v0.5 起每周采集
- 合成：构造 50 个"漂移 vs 没漂移"的对照样本

#### L3.4 · 北极星-主线对齐度评分

**任务**：给定北极星和主线，输出对齐度 0-1。

**输入**：
- 北极星陈述
- 一条主线（含 motivation / criteria）

**输出**：
```typescript
{
  alignmentScore: 0.85,
  reasoning: "主线『转型 PM』直接服务于身份『影响 100 万产品人』",
  warningIfLow?: "主线和北极星没明显关联，要不要修订其中一个？"
}
```

**理想态**：
- ✅ 强对齐（直接服务身份）→ 0.8-1.0
- ✅ 中度对齐（间接相关）→ 0.5-0.8
- ✅ 弱对齐（仅次要意义）→ 0.2-0.5
- ✅ 无对齐 → < 0.2 + 主动 warning
- ❌ 反例：所有主线都打 0.9（无区分度）/ 没有 reasoning

**量化指标**：
- 与人工标注一致率 ≥ 80%（Pearson r ≥ 0.7）
- Warning 触发的精确率 ≥ 70%（避免误报）

**eval 方法**：A（与人工 ground truth 比对）+ B

**数据生产**：
- 合成：200 对（北极星 × 主线）的人工标注对齐度
- 数据格式：(NS, mainline, score 0-1, reasoning)

#### L3.5 · 主线推断（基于观察）

**任务**：观察用户 7-14 天的对话 / 行为，推断 1-3 条候选主线。

**输入**：
- 7-14 天对话 entity / topic / sentiment 流
- 已设定的北极星
- 用户已声明的主线（避免重复）

**输出**：
```typescript
{
  candidateMainlines: [
    {
      title: "转型 PM",
      motivation: "服务北极星「影响 100 万产品人」",
      evidence: ["对话 #234 提到 OKR 5 次", "复盘 #12 写『焦虑做不好需求』"],
      windowEstimate: 30,
      confidence: 0.72
    }
  ]
}
```

**理想态**：
- ✅ 候选与北极星对齐 ≥ 0.6
- ✅ Evidence ≥ 3 条且跨来源
- ✅ 不重复已存在主线（语义去重）
- ❌ 反例：候选是"少看手机"这种琐碎的目标 / Evidence 全来自 1 天

**量化指标**：
- 用户接受候选率 ≥ 30%
- 候选与用户实际新建主线的语义相似度 ≥ 0.7（如果用户接受）
- 误报率（用户拒绝并说"完全不是这个方向"）≤ 15%

**eval 方法**：B + C

**数据生产**：
- 合成：60 个虚拟用户的 14 天行为流 + 人工标"应该推断出哪些主线"
- 真实：v0.5 起 alpha 用户 + 人工标注

#### L3.6 · 主线分解为今日动作

**任务**：给定主线 + 今天日期 + 用户精力曲线，输出 ≤ 3 个今日具体动作。

**输入**：
- 一条主线（progress / 完成标准 / 历史动作）
- 今天日期 + 已完成动作 + 剩余精力 budget
- 用户精力曲线（哪个时段最佳）

**输出**：
```typescript
{
  todayActions: [
    {
      title: "写完支付模块 OKR",
      detail: "2 个案例，参考 v3 模板",
      energyEstimate: 90,
      slot: "上午",
      reason: "你周二上午精力最佳，且这是关键节点"
    }
  ]
}
```

**理想态**：
- ✅ 动作"今天能完成"（不是"启动一个项目"这种粗粒度）
- ✅ 精力评估准确（用户实际完成度 ≥ 80% → 评估 OK）
- ✅ 时间槽与用户精力曲线一致
- ❌ 反例：粒度太粗（"推进主线 A"）/ 时间槽与用户作息冲突 / 一天 5 个动作

**量化指标**：
- 用户接受率（动作进入今日 3 件事）≥ 60%
- 完成率（接受后真完成）≥ 50%
- 粒度评分（人工评 1-5，"恰好今天能做完"）≥ 4

## 4. 数据生产策略

### 4.1 数据分类

| 来源 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| **合成（synthetic）** | 可控、量大、不涉及隐私 | 与真实分布有偏差 | 早期 baseline / 边缘情况 |
| **半合成** | 基于真实模板填充 | 仍需人工校对 | 一级能力的扩量 |
| **真实采集** | 分布真实 | 需用户授权 + 脱敏 | 后期细调 / 罕见场景 |
| **人工 from scratch** | 最高质量 | 最贵 | 关键能力的金标 |

### 4.2 数据生产 playbook

每个 L3 叶子至少一份**金标数据集**（gold dataset）+ **大规模数据集**（broad dataset）：

| 集合 | 量级 | 来源 | 用途 |
|------|------|------|------|
| Gold | 100-500 条 | 人工 from scratch + 3 人独立标注 | 核心 metric 的 ground truth |
| Broad | 5K-50K 条 | 合成 + 半合成 | 训练 / 大规模回归 |
| Live | 流式 | v0.5 起真实用户（授权后）| 持续 fine-tune + 监控 |

### 4.3 标注规范

**多人独立标注**：
- 主观任务（草稿质量、推荐贴切度）→ 3 人独立 + 取众数 / 平均
- 客观任务（意图分类、PII 检测）→ 1 人 + 抽检

**Inter-Annotator Agreement (IAA)**：
- IAA Kappa ≥ 0.7 才算可用
- < 0.7 → 修订 rubric 后重标

**标注 rubric**：
- 每个能力一份独立 rubric.md（不超过 1 页）
- 含：定义、5 分制 / 三档分类的具体描述、≥ 5 个边界例

### 4.4 数据 governance

- **隐私层**：所有真实数据脱敏后才进入数据集
- **版本管理**：Gold dataset 用 DVC 或 Git LFS 管理，每次更新标版本
- **再校准**：每 3 个月对 Gold 抽 10% 重新标注，验证标注漂移

### 4.5 优先级（v0.4 sprint 0 重点先建）

按业务价值 × 可建难度排序：

| 优先级 | 能力 | 理由 |
|--------|------|------|
| **P0** | L3.20 优先级评分、L3.24 意图理解、L3.25 Hint 生成、L3.26 草稿生成、L3.34 推荐决策 | demo 核心循环 |
| **P0** | L3.49 敏感主线脱敏、L3.51 数据分类、L3.52 PII 检测 | 隐私不可妥协 |
| **P1** | L3.12 Identity 提取、L3.13 高频话题、L3.14 观点档案、L3.20 优先级 | 关系画像基础 |
| **P1** | L3.39 复盘反馈、L3.40 长期记忆写入 | 学习闭环 |
| **P2** | 其余 |  |

## 5. 综合评分与上线门禁

### 5.1 单个能力的上线标准

每个 L3 叶子能力都必须满足：

```
能力上线 = (Gold dataset 通过 ≥ X%) AND (Broad dataset 通过 ≥ Y%) AND (隐私 invariant 100%)
```

| 能力类 | Gold X% | Broad Y% | 隐私 |
|--------|---------|----------|------|
| 内容生成（草稿、反馈、推荐）| ≥ 75% | ≥ 70% | 100% |
| 分类 / 检索（意图、优先级）| ≥ 85% | ≥ 80% | 100% |
| 隐私 / 安全（敏感词、PII）| ≥ 99% | ≥ 99% | 100% |

### 5.2 一级能力的综合评分

```
L1 综合分 = Σ (L3 分 × L3 业务权重) / Σ 权重
```

业务权重举例：
- L3.34 推荐决策（高权重，1.0）
- L3.46 时间感知开场（中权重，0.5）
- L3.7 主线冲突检测（中权重，0.5）

### 5.3 版本上线门禁

| 版本 | 阻断器 |
|------|--------|
| v0.4 alpha | P0 能力的 L1 综合分 ≥ 75 + 隐私 100% |
| v0.5 | P0 + P1 能力的 L1 综合分 ≥ 80 + 隐私 100% |
| v0.7 同路人 | L3.43 + L3.49 + L3.50 + L3.51 = 100%（绝对约束）|
| v1.0 发布 | 全部 L1 综合分 ≥ 85 + D30 留存 ≥ 25% + NPS ≥ 30 |

### 5.4 持续监控（v0.4 上线后）

- 每周 Gold + Broad 自动跑一次
- 任意 L3 分数下降 > 5% → 告警
- 用户 NPS 下降 / 投诉关键词触发 → 启动深度 eval
- 每月用户访谈（5-10 人）作为 qualitative 校准
