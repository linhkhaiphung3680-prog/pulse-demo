## 1. 概述

### 1.1 产品定位

**一句话**：让 AI 替你记住关键的人、理解关系、起草回复。

**本质**：不是聊天工具，而是「个人关系操作系统」（Personal Relationship OS）——通讯软件只是它的输入输出端口。

### 1.2 目标用户

| Persona | 痛点 | 核心需求 |
|---------|------|---------|
| **A. 社交超载者**（创业者/PM） | 好友 500+，记不住谁是谁 | 自动沉淀关系画像 |
| **B. 关系经营者**（销售/HR） | 必须主动维护人脉 | 主动提醒 + 起草专业回复 |
| **C. 社交疲惫者**（普通白领） | 不想敷衍但必须回应 | 一键应和 + 节省打字时间 |

v1.0 主攻 Persona A 与 C（消费场景），v1.4 扩展 Persona B（B 端）。

### 1.3 核心价值主张

1. **降低社交关系管理成本**：用户不再靠记忆 / 翻聊天记录
2. **降低社交仪式成本**：应和、确认、客套不再需要手打字

### 1.4 差异化

| 竞品 | 局限 | Pulse 的优势 |
|------|------|------------|
| ChatGPT / Claude | 无人格连续性、无关系记忆、不进入聊天流 | 有名字（小 P）+ 长期偏好图谱 + 嵌入聊天场景 |
| 微信 | 无 AI、无关系画像 | 在微信之上加一层 AI 副驾驶 |
| Monica AI / Clay / Dex | 是 CRM，不进入对话流 | 副驾驶进入聊天起草 + 自然语言对话调教 |
| Notion AI | 文档场景，不懂关系 | 关系记忆 + 上下文 |

**核心壁垒**：长期沉淀的个人偏好图谱——用户用越久，迁移成本越高。

## 2. 用户故事与场景

### 2.1 Persona A 的一天（创业者）

| 时刻 | 场景 | Pulse 介入 |
|------|------|-----------|
| 9:00 早晨 | 打开微信，未读 27 条 | 小 P 早间简报：3 条立刻回 / 2 条可忽略 / 妈妈 30 天未联系 |
| 9:30 通勤 | 投资人 Ava 催 Q3 合同 | 选 hint「承诺时间」→ 草稿「下午 5 点前发你」→ 5 秒发出 |
| 11:00 工作 | 群聊 124 条 | 小 P 摘要：v3 已成基线，你需周五前定 review 时间 |
| 14:00 午饭后 | Mira 邀约 8pm | 跟小 P 商量 → 看到「上次迟到她有失望」→ 采用「热情确认」 |
| 18:00 下班 | Jules 宣布开工作室 | 跟小 P 商量 → 推荐「支持+庆祝」→ 一键采用 |
| 22:00 睡前 | 妈妈发"在干嘛" | 小 P 提示：你倾向 22:00 后回家人，温暖的草稿已就绪 |

### 2.2 Persona C 的关键场景（普通白领）

| 场景 | 体验 |
|------|------|
| **同事生日** | 小 P 主动准备 30 个朋友的生日草稿，每条都不重样（v1.3） |
| **客户反馈** | 草稿区点 ✅ 承诺时间 → 简练专业 → 发出 |
| **家人健康** | 爸爸提复查 → 小 P 看出"他真需要陪同" → 采纳「主动陪同」 |
| **立场分歧** | Noa 发争议话题 → 小 P 提供「温和分歧」+「转移话题」 + 警告"不能敷衍" |

### 3.1 智能社交收件箱（Smart Inbox）

**模块编号**：12（消息优先级评分）

**描述**：用 AI 推断的优先级替代时间排序，让用户打开 App 就知道"应该先回谁"。

#### 功能列表

| ID | 功能 | 状态 | 优先级 |
|----|------|------|--------|
| 1.1 | 三档优先级分组（立刻回 / 稍后回 / 可忽略） | ✓ | **P0** |
| 1.2 | 顶部小 P 早间简报（时间感知） | ◐ | **P0** |
| 1.3 | 每条消息附带"小 P 推荐：附和确认/温和回应/明日再回" | ✓ | **P0** |
| 1.4 | 一键问候（家人 30 天未联系） | ✓ | **P1** |
| 1.5 | 收件箱搜索（按人 / 内容 / 时间） | ○ | **P1** |
| 1.6 | 群消息聚合卡片（"群聊 + 124 条已总结"） | ◐ | **P2** |
| 1.7 | 自定义分组（用户可调整规则） | ○ | **P2** |

#### 输入

- 来源：微信导出包 / iMessage chat.db / WhatsApp backup（v1.1+）
- Signal：发送人 ID、内容、时间戳、群组 ID、@ 标记
- 关联：模块 1（人物画像）、模块 9（关系健康度）

#### 输出

```typescript
{
  buckets: {
    now: [...],      // 立刻回（urgencyScore ≥ 0.8）
    later: [...],    // 稍后回（0.5-0.8）
    ignore: [...]    // 可忽略（< 0.5）
  },
  morningBrief: {
    totalUrgent: 3,
    totalIgnorable: 2,
    keyAlert: "妈妈 30 天未联系"
  }
}
```

#### 验收标准

- **AC-1.1**：Given 用户打开 App / When 加载收件箱 / Then 三档分组在 1 秒内呈现，立刻回数量 ≤ 5
- **AC-1.2**：Given 早晨 6-11 点 / When 打开 App / Then 顶部简报使用早晨语调（"早上好" + 昨晚未看的消息）
- **AC-1.3**：Given 用户对优先级判断不认同 / When 长按消息卡片 / Then 出现"调整优先级"选项，反馈流入模块 7（学习引擎）

### 3.2 聊天 + 副驾驶面板（Chat + Copilot）

**模块编号**：3（Hint 生成）+ 4（Chips 生成）

**描述**：在原生聊天界面下方，固定一个"副驾驶面板"——选 hint → 看草稿 → 调 chip → 发送。

#### 功能列表

| ID | 功能 | 状态 | 优先级 |
|----|------|------|--------|
| 2.1 | 「小 P 看完了」分析摘要 | ✓ | **P0** |
| 2.2 | 2 个动态 hint（基于关系 × 对方话语） | ◐ | **P0** |
| 2.3 | 选 hint 后展开草稿（contenteditable） | ✓ | **P0** |
| 2.4 | 3-4 个场景化 chips（按 hint 类型不同） | ◐ | **P0** |
| 2.5 | 自定义指令输入框（fallback） | ✓ | **P0** |
| 2.6 | 「跟小 P 商量这条」按钮（见 3.3） | ✓ | **P0** |
| 2.7 | 草稿支持 markdown 简单格式（粗体、链接） | ○ | **P1** |
| 2.8 | 草稿支持插入引用（@用户上次说的话） | ○ | **P2** |

#### 关键交互流程

```
用户进入聊天
    ↓
小 P 看完了（intent 摘要 1 行）
    ↓
看到 2 个 hint：[✨ 热情确认] [🤔 小调整]
    ↓
点 hint → 展开草稿 + 显示 3-4 个 chips
    ↓
点 chip → 草稿即时切换（toggle 行为）
    ↓
发送 → 草稿进入聊天 + 触发学习信号（pi-insight）
```

#### 输入

- 当前对方的最新消息
- 对方画像（模块 1）
- 用户偏好图谱（模块 7）
- 历史相似情境的处理结果

#### 输出

```typescript
{
  analysis: { summary: "Mira 期待你确认 8pm 的安排..." },
  hints: [
    { id, emoji, label, sub, draft, chips: [...] },
    { id, emoji, label, sub, draft, chips: [...] }
  ]
}
```

#### 验收标准

- **AC-2.1**：Given 进入任意联系人聊天 / When 滚动到底部 / Then 副驾驶面板可见，2 个 hint 在 800ms 内呈现
- **AC-2.2**：Given 不同关系类型（家人 / 客户 / 朋友） / When 同一句"在吗" / Then 2 个 hint 内容显著不同（lexical similarity ≤ 0.4）
- **AC-2.3**：Given 选中 hint / When 点击其 chip / Then 草稿在 300ms 内切换，原 chip 取消激活
- **AC-2.4**：Given 用户已多次点「更短」chip / When 进入新对话 / Then 默认 chip 排序中「更短」上移到首位

### 3.3 跟小 P 商量这条（Pi Context Sheet）

**模块编号**：2（意图理解）+ 5（推荐决策）

**描述**：从草稿区召唤小 P，弹起底部 sheet，让她以叙事方式说出关系全息分析 + 给出明确推荐 + 支持自由追问。

#### 功能列表

| ID | 功能 | 状态 | 优先级 |
|----|------|------|--------|
| 3.1 | 顶部消息预览卡片 | ✓ | **P0** |
| 3.2 | 4 个结构化分析气泡：意图 / 期待 / 历史 / 风险 | ✓ | **P0** |
| 3.3 | 第 5 个推荐气泡（高/中把握度 + 3 条理由） | ✓ | **P0** |
| 3.4 | 「采用」按钮 → 自动选中推荐 hint | ✓ | **P0** |
| 3.5 | 「辩论一下」按钮 → 小 P 反问 | ✓ | **P0** |
| 3.6 | 推荐气泡的智能适配（已选/未选/选了别的） | ✓ | **P0** |
| 3.7 | 4 个 followup chips（基于关系特定问题） | ✓ | **P0** |
| 3.8 | 自由输入框（继续追问） | ✓ | **P0** |
| 3.9 | 历史对话记录回看（这条消息以前的商量记录） | ○ | **P1** |
| 3.10 | "把这次推荐分享给朋友"（仅匿名版） | ○ | **P3** |

#### 输入

- 当前消息 + 关系画像 + 当前草稿状态
- 历史相似场景的处理记录
- 用户偏好图谱

#### 输出

```typescript
{
  contextPreview: { who, what, when },
  monologue: [
    { type: "intent", content: "..." },
    { type: "expectations", content: [...] },
    { type: "history", content: [...] },
    { type: "risk", content: "..." }
  ],
  recommendation: {
    hintId, verdict, confidence, reasoning,
    ifAgreed, ifDisagreed, debate
  },
  chips: [...],
  qa: { keyword: answer, ... }
}
```

#### 验收标准

- **AC-3.1**：Given 任意联系人 / When 点击"跟小 P 商量这条" / Then sheet 在 400ms 内弹起，4 个分析气泡顺序滑入
- **AC-3.2**：Given 用户未选 hint / When 看到推荐气泡 / Then 文案为"你还没选。我建议——"
- **AC-3.3**：Given 用户已选 Pi 推荐的 hint / When 看到推荐气泡 / Then 文案为"你已选了我推荐的方向" + 优化建议
- **AC-3.4**：Given 用户选了别的 hint / When 看到推荐气泡 / Then 文案为黄色警告 + 折中方案
- **AC-3.5**：Given 点「辩论一下」 / When 按钮点击 / Then 按钮置灰，小 P 追问气泡滑入，输入框自动 focus

### 3.4 人物档案（Person Profile）

**模块编号**：1（个人画像沉淀引擎）

**描述**：每个联系人都有一份 AI 沉淀的档案——关系标签、高频话题、观点立场、重要日期、AI 备忘。

#### 功能列表

| ID | 功能 | 状态 | 优先级 |
|----|------|------|--------|
| 4.1 | 大头像 + 关系标签 chips（老朋友 / 5 年 / Brooklyn） | ✓ | **P0** |
| 4.2 | 关系热度可视化条 | ✓ | **P0** |
| 4.3 | 概览 tab：高频话题 / 重要日期 / 小 P 备忘 | ✓ | **P0** |
| 4.4 | 观点档案 tab：分类的语录 + 立场预警 | ✓ | **P0** |
| 4.5 | 时间线 tab：关键互动事件按时间排列 | ✓ | **P1** |
| 4.6 | 用户可编辑 / 修正 AI 推断 | ◐ | **P1** |
| 4.7 | 用户可标记"重要关系" | ○ | **P1** |
| 4.8 | 联系人合并（同一人在多平台的画像融合） | ○ | **P2** |

#### 输入

- 30 天滑动窗口的对话 + 全量历史索引
- 朋友圈 / 视频号互动 timeline
- 通讯录元数据
- 用户主动反馈

#### 输出

参考 [ROADMAP.md 模块 1](ROADMAP.md#模块-1--个人画像沉淀引擎person-profile-engine) 中的 schema。

#### 验收标准

- **AC-4.1**：Given 任意联系人 / When 进入档案 / Then identity 100% 准确（名字 + 主关系类型）
- **AC-4.2**：Given 联系人有 ≥ 30 天对话历史 / When 看高频话题 / Then 至少 3 个话题被识别，前 2 个用户认可率 ≥ 80%
- **AC-4.3**：Given 系统推断的观点不准 / When 用户长按修正 / Then 数据流入模块 7，下次不再误判
- **AC-4.4**：Given 联系人 ≤ 7 天 / When 看到档案 / Then 显示"小 P 还在了解 TA"占位（避免空状态体验差）

### 3.7 数据接入与隐私（Data Ingestion & Privacy）

**模块编号**：14（隐私 / 可携带）+ 20（合规层）

**描述**：让用户能合规地把现有通讯数据导入 Pulse，并保证数据始终在用户控制之下。

#### 功能列表

| ID | 功能 | 状态 | 优先级 |
|----|------|------|--------|
| 7.1 | 微信对话记录导入（用户主动 export） | ○ | **P0** |
| 7.2 | iMessage 数据接入（macOS 本地 SQLite） | ○ | **P1**（v0.9） |
| 7.3 | WhatsApp chat backup 导入 | ○ | **P1**（v1.1） |
| 7.4 | 通讯录 + 标签接入 | ○ | **P0** |
| 7.5 | 朋友圈截图 OCR 接入 | ○ | **P2** |
| 7.6 | 数据始终本地存储（核心） | ○ | **P0** |
| 7.7 | 端到端加密同步（多设备） | ○ | **P1**（v0.9） |
| 7.8 | LLM 调用：prompt 不留存承诺 | ○ | **P0** |
| 7.9 | 一键导出 .pulse 数据包 | ○ | **P1** |
| 7.10 | 一键删除全部数据 | ○ | **P0** |
| 7.11 | 用户可观测的隐私页面（"小 P 看了什么 / 在云端做了什么"） | ○ | **P1** |
| 7.12 | 本地推理选项（Pro 版） | ○ | **P2**（v0.7） |
| 7.13 | 合规：GDPR / CCPA / PIPL | ○ | **P1** |

#### 验收标准

- **AC-7.1**：Given 用户 export 微信记录 / When 导入到 Pulse / Then 全部对话在 5 分钟内解析完成（10 万条对话）
- **AC-7.2**：Given 启用了多设备同步 / When 在手机端做修改 / Then 服务端无法解密任何数据（zero-knowledge 验证）
- **AC-7.3**：Given 用户点击"删除全部" / When 确认两次 / Then 本地数据 + 云端加密包在 30 秒内全部清除
- **AC-7.4**：Given 任意 LLM 调用 / When 检查 Anthropic API logs / Then prompt 不留存（已在 API 设置中开启 zero-data-retention）
