## 这是什么

Pulse 是一款 AI 社交产品的高保真原型，定位是「**个人关系操作系统**」——通讯软件只是它的输入输出端口。

它解决两个真实痛点：

1. **降低社交关系管理成本**：用户不再靠记忆 / 翻聊天记录管理几百上千人
2. **降低社交仪式成本**：应和、确认、客套不再需要手打字

AI 在产品里的化身叫**小 P**（Pi）——有名字、有时间感、有可被纠正的偏好、有可分层的风格。这是 ChatGPT 类工具型产品做不到的，因为它们没有人格连续性。

## 5 个体验切片

### ① 智能社交收件箱
不再按时间排序，按**小 P 推断的优先级**分组：
- 立刻回（2 条客户/老板/家人紧急）
- 稍后回（3 条朋友日常）
- 可忽略（2 条群消息，淡化显示）

顶部是**小 P 的早间简报**：基于当前时间动态变化（早/中/晚/夜）。

### ② 聊天 + 副驾驶
打开任意联系人，下半屏出现**小 P 看完了**面板：
- 2 个**动态 hint**（基于关系 × 对方话语生成，例如老朋友邀约 → "热情确认 / 小调整"，客户催合同 → "承诺时间 / 请求会议"）
- 选中 hint 后弹出草稿，下方是**场景化 chips**（如热情确认 → 更热情/更矜持/更短/更长），点击即时切换草稿
- 自定义指令（fallback，仅当 chips 不够用）

### ③ 跟小 P 商量这条
草稿区点「跟小 P 商量」打开 sheet——小 P 用自己的语气说出：
- 她到底想说什么（intent）
- 她真正期待（expectations）
- 历史里值得看的（带具体日期的历史信号）
- ⚠️ 风险点
- ⭐ **小 P 的建议**：明确推荐选哪个 hint，附 3 条理由
  - 你已经选了她推荐的 → 给细微优化建议
  - 你选了别的 → 黄色警告 + 折中方案
  - 你点「采用」→ 自动选中 + toast 提示
  - 你点「辩论一下」→ 小 P 反问"你哪里不确定？"

### ④ 关系网络
SVG 同心圆图谱，按关系强度分 3 圈，按社群分 4 色（家人 / 朋友 / 同事 / 客户）。

底部"**关系健康度提醒**"：自动检出需要关注的关系（如妈妈 30 天未联系，带脉冲动画提示）。

### ⑤ 小 P 主面板
底部 tabbar 第 3 位是小 P，进入后 4 个 tab：

- **对话**：跟小 P 自由对话调教（"以后回客户少用 emoji"），她会理解并应答
- **风格**：默认语气 / Emoji 频率 / 直接度 / 长度 4 个三档滑块；底部「针对人群单独设置」可为家人/朋友/同事/客户分别覆盖
- **权限**：读取权限 / 行为权限 / 数据存储三组开关，含锁定的系统级承诺（"数据始终仅在本地"）
- **已学到**：5 项小 P 沉淀的偏好（带置信度进度条），每项可「忘掉」——点击需选择原因（不准确 / 已过时 / 不希望被推断 / 其他），小 P 给出不同的回应

执行关键动作（采纳推荐、连续点 chip、进入 Noa 这种高难度场景）时，会从顶部滑出**紫色"小 P 学到了"浮层**，让"AI 在悄悄学"变成"她在告诉你她学到了什么"。

## 视觉设计

| 元素 | 选择 | 理由 |
|------|------|------|
| 主背景 | `#0B0C10` 暗色 | 让 AI 紫色更突出 |
| AI 主色 | Pulse 紫 `#8B5CF6` | 与微信绿形成对比，让 AI 元素一眼辨认 |
| 自己的消息气泡 | 微信绿 `#95EC69` | 中文用户的肌肉记忆 |
| 对方消息气泡 | 白色 + 尖角 | 严格还原微信 UI |
| 小 P 头像 | 紫色渐变圆形 + 数学符号 π（斜体 serif） | 辨识度极强，单字母可缩放 |
| 设备外壳 | iPhone 边框 + 灵动岛 | 让 demo 像真实 App 而非网页 |

## 技术栈

```
单 HTML 文件
├── Tailwind CSS（CDN，含自定义颜色 token）
├── 内嵌 CSS（动画 / 微信气泡 / 关系图样式）
├── 内嵌 JavaScript（无外部依赖）
│   ├── 数据：CONTACTS / INBOX / AI_REPLIES / PI_CTX_PRESETS
│   ├── 路由：5 屏间切换
│   ├── Pi 模拟：Wizard of Oz（关键词匹配预设响应）
│   └── 关系图：纯 SVG，无 D3
└── 169 KB · 2900 行 · 0 后端
```

**为什么单文件**：让 demo 真正「即开即用」——双击就能本地跑，分享一个 URL 任何人就能体验，0 安装 0 配置 0 后端。

**为什么 Wizard of Oz**：v0.x 阶段我们要解决的是**产品定义问题**而非**技术问题**。先用预设脚本把"理想 AI 体验"做出来，再去想怎么让真实 LLM 达到这个体验——而不是反过来。

## 产品决策的"为什么"

每个 UX 选择背后都有可解释的产品判断，例如：

- 为什么从「四态固定按钮」改为「2 个动态 hint」
- 为什么 chips 是主路径，文字指令只是 fallback
- 为什么「忘掉」必须给原因
- 为什么有了"小 P"就不能再用"AI"作为主语
- 为什么风险点要用黄色而非红色
- 为什么数据"仅在本地"是不可关闭的开关

详见 [PRODUCT_NOTES.md](PRODUCT_NOTES.md)。

## 这个 demo 不做的事

- **真实的微信对接 / 数据导入**：demo 假设我们已有微信全量上下文
- **真实的 LLM 调用**：用 Wizard of Oz 预设脚本模拟
- **用户登录 / 账户系统**
- **桌面端响应式**：移动端 frame 居中即可

这些都在 [ROADMAP](ROADMAP.md) 里，但本 demo 阶段聚焦于「**让产品的形与魂先长出来**」。
